# StripeV2 Monorepo Setup Guide

## Overview

Full-stack TypeScript monorepo built with Turborepo + pnpm workspaces.

| App/Package | Description | Deploy Target |
|---|---|---|
| `apps/web` | Next.js 15 frontend with shadcn/ui | Vercel |
| `apps/server` | Hono + GraphQL Yoga + Pothos API | Railway |
| `apps/docs` | Next.js 15 + MDX documentation | Vercel |
| `packages/db` | Drizzle ORM + PlanetScale PostgreSQL | — |
| `packages/validators` | Shared Zod schemas | — |
| `packages/tsconfig` | Shared TypeScript configs | — |

## Tech Stack

- **Frontend**: Next.js 15 (App Router), React 19, Tailwind CSS v4, shadcn/ui (base-ui)
- **Backend**: Hono, GraphQL Yoga, Pothos (code-first schema)
- **Database**: Drizzle ORM → PlanetScale PostgreSQL
- **Auth**: Better Auth (email/password), server on Hono, client on Next.js
- **Validation**: Zod (shared between frontend and backend)
- **Background Tasks**: BullMQ + Redis (Railway)
- **Cron Jobs**: node-cron (in-process on server)
- **Monorepo**: Turborepo v2 + pnpm v10

## Infrastructure Provisioning

All infrastructure is managed via **Stripe Projects CLI**.

### What was provisioned

```bash
# PlanetScale PostgreSQL (PS-1 single node)
stripe projects add planetscale/postgres --name postgres-db

# Railway hosting (server)
stripe projects add railway/hosting --name railway-hosting

# Railway Redis (for BullMQ)
stripe projects add railway/redis --name redis-cache

# Vercel project (frontend)
stripe projects add vercel/nextjs --name vercel-project
```

### Pull credentials locally

```bash
stripe projects env --pull
```

This writes all provider credentials to `.env`. The key variables populated:

- `PLANETSCALE_URL` — PostgreSQL connection string
- `RAILWAY_HOSTING_URL` — server deployment URL
- `VERCEL_TOKEN`, `VERCEL_PROJECT_ID` — Vercel access

## First-Time Setup

### Prerequisites

- Node.js 22+
- pnpm 10+ (`corepack enable`)
- Stripe CLI with Projects plugin (`stripe projects status`)

### Quick setup

```bash
pnpm setup
```

This runs `scripts/setup.sh` which:

1. Checks prerequisites (pnpm, stripe CLI)
2. Installs dependencies
3. Pulls env vars from Stripe Projects
4. Auto-detects free ports for dev servers
5. Generates `BETTER_AUTH_SECRET` if missing
6. Sets `DATABASE_URL` from `PLANETSCALE_URL`
7. Pushes the DB schema to PlanetScale
8. Builds all packages

### Manual .env additions

After `stripe projects env --pull`, you need these additional variables in `.env`:

```bash
# Auth
BETTER_AUTH_SECRET="<generated by setup script or: openssl rand -base64 32>"
BETTER_AUTH_URL="http://localhost:4001"

# Server
PORT=4001
FRONTEND_URL="http://localhost:3000"

# Frontend
NEXT_PUBLIC_API_URL="http://localhost:4001"

# Database (copy from PLANETSCALE_URL)
DATABASE_URL="<same as PLANETSCALE_URL>"
```

## Local Development

```bash
pnpm dev
```

This runs `scripts/dev.sh` which:

- Loads `.env`
- Auto-detects free ports (avoids conflicts)
- Starts all three apps via Turborepo:
  - **Web**: http://localhost:3000
  - **Server**: http://localhost:4001
  - **Docs**: http://localhost:3001

Local dev hits the **production PlanetScale database** directly.

### Run only the frontend

```bash
pnpm dev:fe
```

### Database commands

```bash
pnpm db:generate   # Generate Drizzle migrations
pnpm db:push       # Push schema to PlanetScale
```

### Auto-deploy on push

- **Web + Docs (Vercel)**: Auto-deploys on push to `master` via GitHub integration. Uses `turbo-ignore` to skip builds when no relevant files changed.
- **Server (Railway)**: Requires manual GitHub connection (see below).

### Diff-based deploys

Each service only rebuilds when its files (or shared package dependencies) change:

- **Vercel**: `ignoreCommand: "npx turbo-ignore"` in each app's `vercel.json`. Turborepo checks if the app or its dependencies changed.
- **Railway**: `watchPatterns` in `railway.json` — only triggers on changes to `apps/server/**`, `packages/db/**`, `packages/validators/**`.

### Environment variables in production

Stripe Projects manages credentials locally but does **not** push them to hosting providers. You must set env vars manually in each dashboard/CLI:

**Railway (server)** — set via `railway variable set`:

```
DATABASE_URL=<PLANETSCALE_URL>
BETTER_AUTH_SECRET=<your secret>
BETTER_AUTH_URL=https://stripev2-production-723a.up.railway.app
FRONTEND_URL=https://stripev2-web-santhoshnarayans-projects.vercel.app
PORT=4000
```

**Vercel (web)** — set via `vercel env add`:

```
NEXT_PUBLIC_API_URL=https://stripev2-production-723a.up.railway.app
```

## Manual Steps (outside CLI)

These steps required manual action in provider dashboards:

### 1. Railway: Connect GitHub for auto-deploys

The Railway CLI doesn't support GitHub connection. Do this in the dashboard:

1. Open https://railway.com → StripeV2 project
2. Click the StripeV2 service → Settings → Source
3. Connect GitHub repo `santhoshnarayan/StripeV2`
4. Root directory: `/`
5. Railway will use the Dockerfile path from `railway.json`

### 2. Vercel: Projects were configured via API

Both Vercel projects (web + docs) were created and configured via CLI + API:

```bash
# Create projects
vercel project add stripev2-web
vercel project add stripev2-docs

# Connect GitHub
vercel git connect https://github.com/santhoshnarayan/StripeV2

# Set root directories and framework via Vercel API
# (CLI doesn't have a settings command for this)
# Web: rootDirectory=apps/web, framework=nextjs
# Docs: rootDirectory=apps/docs, framework=nextjs
```

### 3. Vercel: SSO protection was disabled

Vercel enables SSO deployment protection by default. It was disabled via:

```bash
vercel project protection disable stripev2-web --sso
```

## Key Decisions & Gotchas

- **PlanetScale `sslrootcert`**: PlanetScale's connection URL includes `&sslrootcert=system` which isn't supported. The db package strips it from the URL automatically.
- **Better Auth client exports**: Must use `export const signIn = authClient.signIn` (not destructured) to avoid TypeScript type portability errors with pnpm workspaces.
- **shadcn Button**: Uses `@base-ui/react` and does not support `asChild`. Use `buttonVariants()` with `Link` instead.
- **Server runtime**: Dockerfile uses `tsx` (not compiled `node dist/`) to avoid `.js` extension resolution issues with ESM imports across packages.
- **Port 4000 conflict**: Default server port changed to 4001 for local dev (auto-detected by dev script).
- **Drizzle schema imports**: Use `export * from "./auth"` (no `.js` extension) for drizzle-kit CJS compatibility.
- **Fumadocs abandoned**: Originally planned, replaced with plain Next.js + @next/mdx due to multiple build failures with Fumadocs v15.
